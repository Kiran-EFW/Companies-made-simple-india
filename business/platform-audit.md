## Revenue Stream 3: Services Marketplace — 65% Ready

The catalog and ordering work, but the fulfillment pipeline (the core of the marketplace model) is incomplete.

### What's Built

| Component | Status | Details |
|-----------|--------|---------|
| Service catalog | Done | 50+ services across 6 categories with pricing |
| Service request creation | Done | Users can request services with company context |
| Request payment | Done | Razorpay integration for service payments |
| Upsell recommendations | Done | Context-aware service suggestions based on company stage |
| Admin service request queue | Done | Admin can view/update service request status |
| Service categories | Done | Registration, Compliance, Tax, Legal, Accounting, Amendment |

### Critical Gaps

| Gap | Impact | Why It Matters |
|-----|--------|---------------|
| No CA assignment workflow | Service requests can't be routed to partner CAs | Core marketplace function |
| No fulfillment tracking | No milestones, deliverables, or progress tracking | CAs need to report progress |
| No CA payment settlement | No way to pay CAs their fulfillment fee | Revenue model requires this |
| No CA marketplace dashboard | CAs don't see available service requests | Supply side has no interface |
| No delivery confirmation | No client sign-off on completed work | Quality assurance gap |
| No CA ratings/reviews | No performance tracking for partner CAs | Quality control mechanism |
| No service SLA tracking | No deadline enforcement for CA delivery | Client expectation management |

### Business Model Alignment

- **Service pricing**: Catalog prices match market-rates.md and pricing-strategy.md
- **Platform service margin**: 15-20% is documented but not enforced in code — no split between platform fee and fulfillment fee
- **Service categories**: Match services_catalog.py exactly
- **Upsell logic**: Matches go-to-market.md Stage 5 (contextual service recommendations)

### What Needs to Be Built for Marketplace Launch

1. **Partner CA onboarding** — Register CAs, verify ICAI membership, sign partner agreement
2. **Service assignment engine** — Route requests to available CAs based on specialization, rating, workload
3. **Fulfillment workflow** — Assigned → In Progress → Deliverables Uploaded → Client Review → Completed
4. **Payment settlement** — Calculate fulfillment fee (80-85% of service price), generate payment advice, TDS deduction
5. **CA dashboard** — Available requests, assigned work, earnings, performance metrics
6. **Quality review** — Admin review before client delivery, rejection/revision flow
7. **Rating system** — Client rates CA after service completion

## Conclusion

The platform has exceptional depth for a pre-launch product. The incorporation-to-compliance pipeline is fully built and production-ready. Equity management tools (cap table, ESOP, fundraising) rival dedicated platforms like Carta/Qapita in feature coverage. The CA and investor portals are functional and differentiated.

**The single biggest gap is the services marketplace fulfillment pipeline.** The catalog and ordering work, but the core marketplace mechanics (CA assignment, delivery tracking, payment settlement) don't exist yet. Since marketplace revenue is only ~6-7% of Year 1 revenue per the profitability strategy, this can be built post-launch while using manual coordination (admin assigns work to CAs via ops portal, settles payments offline) for early customers.

**Recommended launch sequence:**
1. Fix the 5 must-have infrastructure items (storage, webhooks, GSTIN, password reset, recurring billing)
2. Launch incorporation + SaaS platform (Revenue Streams 1 + 2)
3. Build marketplace fulfillment pipeline in parallel
4. Launch marketplace (Revenue Stream 3) once 10-15 partner CAs are onboarded
